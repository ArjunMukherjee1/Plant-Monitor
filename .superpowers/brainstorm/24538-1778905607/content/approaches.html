3 Architecture Approaches

Pick the one that fits your comfort level and how much complexity you want to maintain

A

Flutter + Direct HA API (simplest)

The app talks directly to Home Assistant's built-in REST & WebSocket API. All charts and predictions are computed on-device. No extra server to run.

Pros

  • Nothing to deploy beyond the app
  • Uses HA's existing history DB
  • Real-time updates via WebSocket

Cons

  • Push notifications require the app to be open (or a separate HA automation)
  • Predictions limited to what fits on-device
B

React Native + Direct HA API (JS ecosystem)

Same direct-to-HA approach, but built with React Native + TypeScript. Better choice if you already know JavaScript. Larger ecosystem of charting libraries.

Pros

  • JS/TS — very common skill set
  • Rich charting libs (Victory, Recharts)
  • Can share code with a future web version

Cons

  • Same push notification limitation as A
  • React Native can be fiddly on native modules
C

Flutter + Lightweight Backend (recommended)

A small Python/Node backend runs alongside Home Assistant (same machine or a HA add-on). It polls HA, stores richer history, runs predictions, and sends push notifications via FCM. The Flutter app talks to this backend.

Pros

  • Reliable push notifications (even when app is closed)
  • Richer time-series history & predictions
  • Ready for irrigation control later
  • Flutter = beautiful charts & animations

Cons

  • One more service to run (but lightweight)
  • Slightly more setup upfront

⭐ My recommendation: C — push notifications while the app is closed is nearly essential for a plant monitoring app, and the backend slots in cleanly for future irrigation control.
